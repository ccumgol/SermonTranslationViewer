# 개발 트러블슈팅 기록 (교육용)

이 문서는 `translateViewer` 를 만들면서 실제로 만난 에러와 그 해결 과정을
정리한 것입니다. 각 사례는 **증상 → 원인 → 해결 → 교훈** 순서로 기록했습니다.
실시간 음성/번역 시스템을 처음 만드는 사람을 위한 학습 자료로 사용하세요.

---

## 1. 인증 / 계정 문제

### 1-1. 조직(회사·학교) 계정에서 API 키 발급 거부
- **증상**: API 키를 만들려 하면 *"API 키가 허용되지 않습니다. 대신 애플리케이션
  기본 사용자 인증 정보(ADC)를 사용하세요"*
- **원인**: Google Workspace(조직) 계정에 **API 키 발급을 금지하는 보안정책**이 걸려 있음.
- **해결**: 정책이 없는 **개인 Google 계정**으로 키 발급.
- **교훈**: 조직 계정은 보안정책으로 막혀 있는 경우가 많다. 개인 프로젝트는
  처음부터 개인 계정으로 시작하면 우회 작업이 줄어든다.

### 1-2. ADC/Vertex 경로로 우회하려 했으나 모델이 없음
- **증상**: 조직 계정을 유지하려고 ADC(Vertex AI) 경로를 검토했으나, 우리가 쓰는
  `gemini-3.5-live-translate-preview` 가 Vertex 에 없었음.
- **원인**: 이 모델은 당시 **Gemini Developer API(AI Studio, API 키 방식) 전용**.
- **해결**: 개인 계정 + API 키 경로로 확정.
- **교훈**: "우회 경로"가 항상 같은 모델을 제공하지 않는다. **모델 가용성을 먼저
  확인**하고 인증 방식을 정해야 한다. (공식 문서·포럼으로 검증)

### 1-3. 브라우저 자동화로 프로젝트/키 생성 시 "의심스러운 요청" 차단
- **증상**: 자동화로 새 프로젝트/키를 만들면 *"The request is suspicious. Please try again."*
- **원인**: Google 의 **봇 탐지**가 자동화된 생성 요청을 차단.
- **해결**: 프로젝트·키 생성은 **사람이 직접** 수행. (자동화는 조회·확인까지만)
- **교훈**: 보안 관련 생성 작업은 봇 탐지에 막힌다. 우회하려 하지 말고 수동으로 처리한다.

## 2. 모델 / API 설정 문제

### 2-1. 세션은 연결되는데 오디오 전송 시 `1007 invalid argument`
- **증상**: `[live] 세션 연결됨` 직후 `1007 None. Request contains an invalid argument.`
  가 2초마다 반복되며 자막이 안 나옴.
- **원인**: 목표 언어 코드를 **`en-US`(지역 접미사 포함)** 로 넘김. 이 모델은
  지역 접미사 붙은 코드를 거부함.
- **해결**: **짧은 코드**(`en`, `ja`, `es` …)로 변경. 단 중국어는 `zh`, `zh-CN` 둘 다 허용.
- **검증 방법**: 무음 PCM 을 보내며 여러 코드를 실제로 연결 테스트 → 통과/거부 표로 확정.
- **교훈**: 연결은 setup 검증, **실제 invalid argument 는 첫 데이터 전송 때** 드러난다.
  코드값 같은 사소한 포맷이 전체를 막을 수 있으니 **추측 말고 실측**한다.

### 2-2. 다국어를 한 세션에서 못 냄
- **증상**: 한 세션이 한 언어만 출력.
- **원인**: 모델 구조상 세션 = 목표 언어 1개.
- **해결**: 언어마다 **별도 세션**을 띄우고, 하나의 오디오를 **fan-out** 으로 복제해 공급.
- **교훈**: 동시 다국어는 곧 동시 세션 = **비용·동시성 N배**. 사용자에게 비용을 명시.

## 3. 결제(Billing) 문제

### 3-1. `1011 prepayment credits are depleted`
- **증상**: 키·모델은 정상인데 `Your prepayment credits are depleted`.
- **원인**: 선불(prepaid) 결제 프로젝트의 **크레딧 소진**.
- **해결**: 크레딧 충전(또는 후불 전환). **결제수단 입력은 반드시 본인이 직접.**
- **교훈**: 연결이 결제 단계까지 갔다는 건 인증·권한은 정상이라는 뜻. 에러 코드를
  계층별로 구분하면(인증 vs 권한 vs 결제) 원인 추적이 빠르다.

## 4. 환경 / 실행 문제

### 4-1. `ModuleNotFoundError: No module named 'sounddevice'`
- **증상**: `python3 server/audio_input.py` 실행 시 모듈 없음.
- **원인**: **가상환경(.venv)을 활성화하지 않고** 시스템 파이썬으로 실행.
- **해결**: `source .venv/bin/activate` 후 실행. (`./start.sh` 가 이를 자동 처리)
- **교훈**: 의존성은 venv 안에만 설치된다. 실행 스크립트로 활성화를 강제하면 실수가 준다.

### 4-2. 오디오 장치 인덱스가 문자열로 처리됨
- **증상**: `AUDIO_INPUT_DEVICE=2` 로 했는데 엉뚱한 장치가 잡힘.
- **원인**: `.env` 값은 항상 문자열 → sounddevice 가 "이름이 2인 장치"를 찾으려 함.
- **해결**: 숫자면 `int`(인덱스), 아니면 `str`(이름)로 변환하는 파서 추가(`_parse_device`).
- **교훈**: 외부 입력(.env)은 항상 문자열. **경계에서 타입을 정규화**한다.

## 5. UI / 자막 표시 문제

### 5-1. 자막이 너무 짧게 깜빡임
- **증상**: 누적 문장이 아니라 방금 들어온 짧은 조각만 깜빡임.
- **원인**: 전사가 **조각(delta)** 으로 오는데, 매번 화면을 그 조각으로 통째 교체.
- **해결**: 조각을 **누적**해 문장을 만들고, 직전 문장도 함께 유지(rolling).
- **교훈**: 스트리밍 텍스트는 "증분"이다. 누적/교체 정책을 명확히 설계한다.

### 5-2. 자막이 위·아래로 반쯤 잘림
- **증상**: 행이 나뉠 때 글자가 행 경계에서 절반씩 잘림.
- **원인**: 행을 `justify-content: center`(세로 가운데)로 둬서 넘침이 **양쪽**으로 잘림.
- **해결**: 텍스트를 행 **하단에 절대 고정**(`position:absolute; bottom:0`)하고 위쪽만 잘리게.
- **교훈**: "최신 내용이 항상 보여야 하는" UI 는 **하단 고정 + 위쪽 클리핑**이 정석.

### 5-3. 송출 화면에 브라우저 메뉴/주소창이 같이 나감
- **증상**: 프로젝터에 탭·주소창이 보임.
- **해결**: 별도 팝업 창으로 열기 + **전체화면 토글**(Fullscreen API, 더블클릭).
- **교훈**: 송출은 OBS 브라우저 소스 또는 전체화면이 가장 깔끔.

## 6. 오프라인(로컬) 백엔드 문제 — 가장 많은 학습 포인트

### 6-1. 종료 시 `RuntimeError: Event loop is closed` traceback
- **증상**: `Ctrl+C` 로 끄면 정상 종료 후 traceback 출력.
- **원인**: STT 추론을 `asyncio.to_thread`(기본 스레드풀)로 돌리는데, 루프가 닫힌 뒤
  그 스레드풀이 닫힌 루프를 건드림.
- **해결**: STT 추론을 **전용 `ThreadPoolExecutor`** 로 분리하고, 종료 시
  `aclose()` 로 태스크·스레드풀을 명시적으로 정리.
- **교훈**: 블로킹 작업을 기본 풀에 남기면 종료 race 가 난다. **수명주기를 직접 관리**한다.

### 6-2. STT 가 성경 용어 목록을 그대로 출력(환각)
- **증상**: 말한 내용과 무관하게 *"여호와 하나님 예수 그리스도 …"* 만 반복 표시.
- **원인**: 인식률을 높이려고 STT `context` 에 **긴 고유명사 목록**을 넣었더니,
  모델이 그 목록을 전사 결과로 토해냄.
- **해결**: 기본 컨텍스트를 **빈 문자열**로. 필요하면 소수 핵심 용어만 짧게.
- **교훈**: ASR 의 context/hotword 기능은 **과하게 쓰면 환각**을 부른다. 적게 쓴다.

### 6-3. 실시간 스트리밍 STT 의 인식률이 극악
- **증상**: 라이브에서 `하면.이미.어.이미.이렇게…` 같은 **필러 조각**만 나옴.
- **원인**: Qwen3-ASR 의 **스트리밍 모드**가 오디오를 누락·왜곡(전체파일 전사는 거의 완벽).
- **해결**: 스트리밍을 버리고 **VAD(침묵 감지)로 발화 구간을 잘라** 그 구간을
  `transcribe()` 로 통째 처리 → 전체파일급 정확도.
- **검증**: 같은 음성을 (a) 스트리밍 (b) VAD 분할로 돌려 결과를 직접 비교.
- **교훈**: "실시간 = 스트리밍"이 항상 정답은 아니다. **짧은 지연을 감수하고
  발화 단위로 정확히** 처리하는 편이 자막 품질엔 유리하다.

### 6-4. 첫 문장 번역이 ~18초 지연
- **증상**: 첫 번역만 매우 느림.
- **원인**: 로컬 번역 모델(Ollama)의 **콜드 로딩**.
- **해결**: 서버 시작 시 더미 번역으로 **예열(warmup)** + `keep_alive` 로 웜 유지.
- **교훈**: 로컬 LLM 은 첫 호출이 느리다. 미리 메모리에 올려둔다.

### 6-5. 단어마다 번역 호출로 지연 누적
- **증상**: 말할수록 번역이 점점 밀림.
- **원인**: 전사 **조각마다** 무거운 번역 모델을 호출 → 큐 적체.
- **해결**: **문장/발화 단위**로 모아 번역(호출 수 급감). VAD 도입 후엔 발화 구간 = 1회 호출.
- **교훈**: 무거운 후처리는 **호출 빈도**가 곧 지연. 의미 단위로 묶는다.

### 6-6. 더 큰 STT 모델이 더 낫지 않음
- **증상**: 정확도를 위해 1.7B 로 키웠더니 **느리기만 하고 더 정확하지 않음**.
- **해결**: 0.6B 유지(웜에서 9초 음성을 ~3.7초에 처리). 모델 크기는 env 로 교체 가능하게.
- **교훈**: "큰 모델 = 무조건 좋음"이 아니다. **측정으로** 비용 대비 효과를 본다.

## 7. Git / 협업

### 7-1. 푸시 거부 (non-fast-forward)
- **증상**: `git push` 시 *"Updates were rejected … use 'git pull' before pushing"*
- **원인**: 원격에 로컬에 없는 커밋(웹에서 수정한 README)이 있었음.
- **해결**: `git pull --rebase origin main` 후 푸시. (동일 변경은 자동 skip)
- **교훈**: 원격을 다른 경로(웹 UI 등)로도 바꾸면 분기된다. **푸시 전 pull --rebase** 습관화.

## 부록: 디버깅에 쓴 일반 원칙

1. **추측보다 실측**: 코드값·포맷 문제는 작은 재현 스크립트로 직접 테스트해 확정.
2. **에러를 계층으로 구분**: 인증 / 권한 / 결제 / 데이터(invalid argument) 를 나눠 본다.
3. **공식 문서로 검증**: 학습 시점 이후의 모델/스펙은 반드시 공식 문서·포럼으로 확인.
4. **민감정보 분리**: API 키·로그·원고는 `.gitignore` 로 처음부터 격리.
5. **되돌릴 수 있게**: 큰 변경은 **브랜치**에서. 동작 버전(main)을 항상 보호.
